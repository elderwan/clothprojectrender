Cart / Order Flow05

Goal / Explain the shopping loop clearly

Step 1

Users select a size on the detail page and add the item to the cart, or save it in the wishlist first and move it later.

→
Step 2

The cart supports quantity updates, item removal, and live recalculation of subtotal, shipping, and total.

→
Step 3

When the order is confirmed, a Supabase RPC creates the order, inserts order_items, deducts stock, and clears the cart.

Shopping CartQuantity Edit / Price Summary

The strongest part of the order flow is inventory consistency.

  • `create_demo_order_with_inventory` validates stock first, then creates the order, then deducts inventory in one backend flow.
  • After creation, the order enters `processing`, and the client-side order detail page completes address confirmation and demo payment confirmation.
  • The order detail page shows item lines, shipping address snapshot, tax, and status instead of only showing a success message.
  • The admin side also enforces allowed status transitions from pending to delivered to avoid arbitrary updates.

This is the best slide to prove that the business loop is already complete. Once a product enters the cart, the order, stock, payment confirmation, and admin status workflow are all connected.

Architecture08

Goal / Explain the implementation structurally

The codebase uses a layered structure: controllers handle pages, services handle business logic, and models handle Supabase data access.

Presentation Layer
  • `views/client/*` and `views/admin/*` are rendered with EJS templates.
  • The UI uses Tailwind CDN plus a small amount of native browser JavaScript.
  • The pages feel like a static fashion site on first view, but the data is injected by the backend.
Business Layer
  • `controllers` coordinate routes and rendering.
  • `services` encapsulate auth, cart, order, and product recommendation logic.
  • Middleware centralizes user injection and admin permission checks.
Data Layer
  • `@supabase/supabase-js` connects the application to the cloud database.
  • The model layer manages products, orders, cart, wishlist, and related tables independently.
  • Some multi-step operations are handled through Supabase RPC and SQL functions.
The frontend is responsible for looking like a fashion brand site. The backend is responsible for acting like a real commerce system.

Database Design09

Goal / Show how the data model supports the business

The database is not just a product table. It is divided into multiple entities around the full transaction journey.

Product Domain`categories`, `products`, and `product_images` support categorization, stock, primary image selection, multi-image display, and audience tags.
User Domain`users` and `user_addresses` support identity, role, contact information, and default address behavior.
Transaction Domain`cart_items`, `orders`, and `order_items` support the full path from cart to order placement to order tracking.
Preference Domain`wishlist_items` supports delayed purchase behavior by letting users save items before buying.
Content Operations`home_banners` makes homepage and category visuals configurable for campaigns and seasonal updates.
Functions & TriggersThe database defines `update_updated_at`, `increment_product_click_count`, and the order-inventory RPC for consistency.
Most Important PointOrder creation and stock deduction are handled atomically at the database side

Completed vs Pending10

Goal / Clarify what is truly completed

The project already supports a full demo, but it still has some course-project boundaries.

Completed

  • The client pages, admin pages, routes, middleware, and database tables are connected.
  • Product browsing, detail pages, cart, orders, address book, and wishlist work in the running project.
  • Admins can manage products, orders, users, banners, and categories.
  • The current repository passes `npm run build` successfully.

Current Limits

  • The payment step is demo-style `confirm-pay`, not a real payment gateway integration.
  • The repository includes Jest configuration, but the project itself does not yet include real test coverage.
  • The sorting options include `sales_desc / sales_asc`, but the model layer does not fully implement sales-based sorting.
  • The cart currently assumes a logged-in user. A full guest cart flow is not finished yet.

This slide should be presented proactively. If your instructor asks about real payment or guest checkout, you can answer clearly: the current version focuses on a complete course-level business loop, while production-level capabilities are the next step.

Highlights & Challenges11

Technical Highlights

  • Express plus EJS makes server-side rendered pages straightforward and works well for a brand-style course project.
  • A custom HMAC-signed JWT cookie approach keeps authentication from relying only on session state.
  • Click counting, revenue statistics, and inventory deduction are pushed into database functions to reduce scattered backend logic.
  • Cloudinary upload support and multi-image management move product maintenance closer to a real store workflow.

Challenges and Solutions

  • Challenge 1: A fashion store needs a strong visual identity. The solution was a unified black-white-gray palette, serif headlines, large imagery, and minimal interactions.
  • Challenge 2: Order placement must avoid inventory mismatch. The solution was a database-side RPC for validation, order creation, and stock deduction.
  • Challenge 3: Building both client and admin modules can become messy. The solution was layered separation with routes, controllers, services, and models.
  • Challenge 4: Products need more than one image. The solution was a dedicated `product_images` table with ordering and primary-image support.
If this project were pushed one step further, the most valuable upgrades would be real payment integration, guest cart support, automated tests, and deeper operational analytics.
